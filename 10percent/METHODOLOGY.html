1. Overview

This dashboard provides a comprehensive analysis of household wealth and corporate power across seven major U.S. metropolitan areas: Los Angeles, New York, Chicago, Dallas, Houston, Miami, and San Francisco. The analysis identifies the top 10% wealthiest ZIP codes within these 7 metropolitan areas for both households and corporations, creating comparable metrics that integrate multiple dimensions of economic power.

1.1 Geographic Scope and Limitations

Important: This analysis is limited to 7 specific major metropolitan areas. The "Top 10%" rankings are relative to the universe of ZIP codes within these 7 metros, not the entire United States.
  • Included Metro Areas: Los Angeles, New York, Chicago, Dallas, Houston, Miami, San Francisco
  • Not Included: Other wealthy metropolitan areas such as Boston, Washington DC, Seattle, Philadelphia, Atlanta, and others are not part of this analysis
  • Total ZIP Codes Analyzed: 2,716 ZIPs for households, 3,598 ZIPs for corporate data
  • Top 10% Threshold: Calculated separately for households (272 ZIPs) and corporations (360 ZIPs) within the 7-metro universe

The methodology combines:

  • Household Wealth: IRS tax return data, Census demographics, and geographic accessibility
  • Corporate Power: U.S. Census Bureau business patterns, employment, revenue estimates, and industry concentration
  • Geographic Analysis: Travel times to major airports, geodesic distances, and regional patterns

2.4 Data Harmonization & Missing Data Treatment

ZIP Code vs ZCTA Boundaries

This analysis uses ZIP Code Tabulation Areas (ZCTAs) from the U.S. Census Bureau TIGER/Line shapefiles. ZCTAs are statistical approximations of ZIP code delivery areas created by the Census Bureau. Important notes:

  • ZCTAs approximate but do not exactly match USPS ZIP code delivery areas
  • Some ZIP codes may not have corresponding ZCTAs (e.g., PO boxes, military addresses)
  • ZCTAs are updated every 10 years with the decennial census (current analysis uses 2020 ZCTAs)
  • IRS SOI data uses actual ZIP codes, while Census ACS uses ZCTAs - matching is done by ZIP code string

Census Bureau Data Suppression

The Census Bureau suppresses detailed employment data by industry for privacy protection when there are too few establishments or when disclosure would reveal individual business information. Our treatment:

  • Suppressed Employment: We estimate employment by industry proportionally: Emp_industry = Emp_total × (Estab_industry / Estab_total)
  • Assumption: This assumes establishments within an industry are similar in size, which may not hold for ZIPs with very large or very small businesses
  • Total Employment: Available for NAICS code '00' (all industries combined) and is 100% real data

Missing ZIP Codes

ZIP codes may be excluded from analysis for several reasons:

  • No tax return data in IRS SOI (e.g., PO boxes, military addresses)
  • No Census ACS demographic data available
  • No business establishments in Census CBP
  • ZIP code not in the 7-metro geographic boundaries

Coverage: The analysis includes 2,716 ZIPs for household analysis and 3,598 ZIPs for corporate analysis within the 7 metropolitan areas. ZIPs outside these metros or with missing critical data are excluded.

Google Distance Matrix API Configuration

Travel time calculations use the following configuration:

  • Departure Time: "now" - Uses real-time traffic conditions at the moment of API call
  • Origin: ZIP code centroid (geographic center) from TIGER/Line shapefiles
  • Destination: Major airport coordinates for each metro area
  • Mode: Driving (automobile)
  • Batch Processing: 25 origins per API request (API limit)
  • Caching: Results cached in JSON files to avoid redundant API calls
Note: Travel times reflect conditions at the time of API call and may vary significantly by time of day, day of week, and season. Results represent a snapshot, not an average over time.

4. Corporate Power Analysis

4.1 Data Processing

Corporate data processing involves several steps due to Census Bureau privacy protections:

  1. Data Fetching: Parallel API calls to Census CBP endpoint for all ZIP codes
  2. Employment Estimation: Since Census suppresses employment by industry, we estimate it proportionally:
    Employment_industry = Employment_total × (Establishments_industry / Establishments_total)
  3. Revenue Estimation: Revenue is estimated using industry-specific revenue-per-employee ratios from BLS/Census benchmarks. The implementation uses a simplified two-tier model:
    Revenue_Total = Employment_total × $200,000 (average)
    Revenue_Power = Power_Employment × $350,000 (power industries average)

    However, the codebase includes detailed NAICS-specific coefficients (shown in table below) that could be used for more granular estimation.

  4. Industry Classification: Industries are classified using NAICS 2-digit codes

5. Geometric Score Calculation (Households)

The Geometric Score is a composite metric that combines multiple dimensions of household wealth using a weighted geometric mean. This approach ensures that all components contribute meaningfully and prevents any single dimension from dominating the score.

5.1 Normalization

Each variable is first normalized to a [0,1] scale using global bounds (across all cities) to ensure comparability:

Normalized_Value = (Value - Min_Global) / (Max_Global - Min_Global)

After normalization, each variable is raised to the power of its weight in the geometric mean calculation.

5.2 Component Variables

Component Weight Description
IRS_Norm 50% Normalized AGI per return (proxy for individual wealth)
Time_Squared 20% Normalized travel time to airport, squared (exclusivity indicator)
Model Design Choice - Distance/Time Component: This methodology uses a normative model assumption that values exclusive, secluded enclaves away from high-traffic airport areas. ZIP codes with longer travel times to airports receive higher scores in this component. This is a design decision, not an empirical finding. ZIP codes that are wealthy but located near airports may be undervalued by this component. Alternative models could value accessibility instead, or use a U-shaped function that penalizes both very close and very far distances.
Pop200k_Norm 20% Normalized count of households earning $200k+ (concentration) Density_Norm 10% Normalized density of high-income households (spatial concentration)

5.3 Geometric Mean Formula

The Geometric Score is calculated as a weighted geometric mean. To prevent zeros from eliminating scores entirely, a small epsilon (ε = 1e-10) is added to each normalized component before exponentiation:

Geometric_Score = ((IRS_Norm + ε)^0.50) × ((Time_Squared + ε)^0.20) × ((Pop200k_Norm + ε)^0.20) × ((Density_Norm + ε)^0.10)

Where:

  • Time_Squared = (Time_Norm)² - The normalized travel time is squared before being raised to the 0.20 power, amplifying the effect of distance/exclusivity
  • ε = 1e-10 - A very small constant added to prevent zero values from making the entire product zero
Why Geometric Mean? The geometric mean ensures that all components must be present for a high score. A ZIP code with extremely high AGI but zero high-income households would score low, reflecting the multi-dimensional nature of wealth.
Zero Handling: ZIP codes with values equal to the global minimum in any component will have normalized values near zero, resulting in very low scores (but not exactly zero due to epsilon). This is intentional: the model requires all dimensions to be present for a high score. ZIPs with minimum values in critical components are effectively excluded from top rankings.

Note on Pop200k vs Density: While these variables are correlated, they capture different aspects: Pop200k_Norm measures absolute count (total high-income households), while Density_Norm measures spatial concentration (households per square kilometer). Together they capture both scale and intensity of wealth concentration.

5.4 Top 10% Selection

The top 10% richest ZIP codes are identified using the 90th percentile threshold:

Threshold_90 = Quantile(Geometric_Score, 0.90)

All ZIP codes with Geometric_Score ≥ Threshold_90 are included in the top 10% analysis.

6. Corporate Score Calculation

The Corporate Score is designed to be comparable to the Household Geometric Score, using a similar weighted geometric mean approach. It integrates revenue, employment, power industry concentration, and geographic accessibility.

6.1 Component Variables

Component Weight Description
Revenue_Norm 35% Normalized estimated revenue (millions $)
Employment_Norm 30% Normalized total employment
PowerShare_Norm 15% Normalized power industries employment percentage
Distance_Squared 20% Normalized distance to airport, squared (exclusivity indicator)
Model Design Choice - Distance Component: Similar to the household analysis, this methodology uses a normative assumption that values corporate activity spread across the metro area, not just concentrated near airports. ZIP codes with greater distances to airports receive higher scores. This reflects a design choice to identify "spread" rather than "concentration" patterns. Note that corporate power often concentrates near airports and CBDs; this model intentionally favors a different pattern. Sensitivity analysis could test alternative weightings or functional forms.

6.2 Corporate Score Formula

Corporate_Score = (Revenue_Norm^0.35) × (Employment_Norm^0.30) × (PowerShare_Norm^0.15) × (Distance_Squared^0.20)

Where Distance_Squared = (Distance_Norm)², similar to the household analysis.

Note on Corporate_Power_Index: A separate metric called Corporate_Power_Index is also calculated for statistical analysis. It uses an arithmetic weighted average (40% Revenue + 30% Employment + 30% Power Share) without the distance component. This index is used in intersection analysis and statistical comparisons, but not for Top 10% selection (which uses Corporate_Score with geometric mean).

6.3 Distance Calculation

Distance to airport is calculated using:

  1. Primary Method: Travel time from Google Distance Matrix API, converted to distance assuming 40 km/h average speed
  2. Fallback Method: Haversine formula (geodesic distance) if travel time is unavailable
Distance_km = (Travel_Time_Min / 60) × 40

Or, if travel time unavailable:

Distance_km = Haversine(ZIP_centroid, Airport_coordinates)

8. Geographic Analysis

8.1 Travel Time Calculation

Travel times are calculated using Google Distance Matrix API. See Section 2.4 for detailed configuration. Key parameters:

  • Mode: Driving (automobile)
  • Departure Time: "now" (real-time traffic conditions at API call time)
  • Origins: ZIP code centroids (geographic center from TIGER/Line shapefiles)
  • Destinations: Major airport coordinates for each city
  • Batch Size: 25 origins per API request (API limit)
  • Parallel Processing: 30 workers for corporate data, sequential for household data
  • Caching: Results cached to ensure reproducibility and reduce API costs

Important: Travel times represent a snapshot of conditions at the time of API call, not an average over time. Results may vary by time of day, day of week, and season.

8.2 Geodesic Distance

Geodesic (great-circle) distance is calculated using the Haversine formula:

a = sin²(Δlat/2) + cos(lat1) × cos(lat2) × sin²(Δlon/2)
c = 2 × atan2(√a, √(1-a))
Distance = R × c

Where R = 6,371 km (Earth's radius)

8.3 Weighted Distance Analysis

For each city, we calculate several distance metrics:

  • Weighted Distance: Distance weighted by travel time (longer travel = more weight)
  • Mean/Median Distance: Simple statistics of distances
  • Region Spread: Maximum distance between any two top 10% ZIP codes
  • Bounding Radius: Approximate radius of the region containing all top 10% ZIPs
  • Centroid Distance: Distance from airport to the centroid of all top 10% ZIPs

8.4 Speed Calculation

Average speed to airport is calculated as:

Speed_kmh = Distance_km / (Travel_Time_Min / 60)

This metric helps identify areas with good highway access vs. congested urban routes.

9. Statistical Analyses

9.1 Weighted Averages

Weighted averages are calculated to give more importance to ZIP codes with higher economic activity:

Household Weighted Averages

  • Speed Weighted by Wealth:
    Weighted_Speed = Σ(HH200k × AGI × Speed) / Σ(HH200k × AGI)
  • AGI Weighted by HH200k+:
    Weighted_AGI = Σ(HH200k × AGI) / Σ(HH200k)

Corporate Weighted Averages

  • Power Index Weighted by Employment:
    Weighted_Power = Σ(Power_Index × Employment) / Σ(Employment)
  • Revenue Weighted by Employment:
    Weighted_Revenue = Σ(Revenue × Employment) / Σ(Employment)

9.2 Time Weighted by Wealth

For corporate analysis, travel time is weighted by corporate wealth (revenue × employment), similar to how household analysis weights by (HH200k × AGI):

Weighted_Time = Σ(Time × Corporate_Wealth) / Σ(Corporate_Wealth)

Where Corporate_Wealth = Revenue × Employment

9.3 Power Industries Analysis

Power industries are analyzed by:

  • Employment Share: Percentage of total employment in power industries
  • Revenue Share: Percentage of total revenue from power industries
  • Establishment Count: Number of power industry establishments

9.4 Comparative Statistics

Various scatter plots and correlations are calculated:

  • Revenue vs. Employment (correlation analysis)
  • Power Index vs. Power Share (industry concentration)
  • Employment vs. Establishments (business density)
  • Revenue per Employee (productivity metric)

11. Intersection Analysis

Intersection analysis identifies ZIP codes that appear in both the top 10% household and top 10% corporate rankings.

11.1 Intersection Calculation

The intersection is defined as the set of ZIP codes that appear in both top 10% rankings:

Intersection = {ZIP codes where Geometric_Score ≥ Threshold_HH AND Corporate_Score ≥ Threshold_Corp}

11.2 Intersection Metrics - Explicit Formulas

Three key metrics are calculated to measure the overlap between household and corporate top 10%:

% of City = |A ∩ B| / |Total City ZIPs| × 100

% of HH Top 10% = |A ∩ B| / |A| × 100

% of Corp Top 10% = |A ∩ B| / |B| × 100

Where:

  • A = Set of ZIP codes in Household Top 10%
  • B = Set of ZIP codes in Corporate Top 10%
  • A ∩ B = Intersection (ZIPs in both sets)
Statistical Context: Under random independence, if 10% of ZIPs are in Household Top 10% and 10% are in Corporate Top 10%, the expected intersection would be approximately 1% (10% × 10%). Observed intersections significantly higher than 1% indicate positive correlation between household wealth and corporate power locations.

11.3 Combined Score (for Intersection ZIPs)

For ZIP codes in the intersection, a Combined Score is calculated as a simple weighted average:

Combined_Score = 0.5 × Geometric_Score + 0.5 × (Corporate_Power_Index / 100)

Where:

  • Geometric_Score: Household wealth score (geometric mean, see Section 5)
  • Corporate_Power_Index: Corporate power metric (arithmetic weighted average, see note below)
Important Distinction:
  • Corporate_Score (used for Top 10% selection): Geometric mean with 4 components including distance (Section 6)
  • Corporate_Power_Index (used for statistical analysis and Combined_Score): Arithmetic weighted average with 3 components (40% Revenue + 30% Employment + 30% Power Share), without distance component
The Combined_Score uses Corporate_Power_Index (not Corporate_Score) to combine with Geometric_Score.

11.4 Additional Metrics

  • Jaccard Index: J = |A ∩ B| / |A ∪ B| - Measures similarity between the two sets
  • Correlation: Pearson correlation coefficient between Geometric Score and Corporate Score

11.3 Results

Current Statistics:

  • Household Top 10%: 272 ZIPs
  • Corporate Top 10%: 360 ZIPs
  • Intersection: 197 ZIPs (72.4% of household top 10%)
  • Correlation: 0.261 (positive but moderate correlation)
Interpretation: A 72.4% overlap suggests that wealthy households and powerful corporations tend to co-locate, but not perfectly. The moderate correlation (0.261) indicates that while there is a relationship, other factors also play important roles.

12. Data Estimates & Limitations

12.1 What is Direct from Government Sources?

Direct Government Data (100% Real Counts):
  • All establishment counts (Census CBP)
  • Total employment per ZIP code (Census CBP, NAICS '00')
  • Total annual payroll (Census CBP)
  • Household income data (IRS SOI)
  • Population and demographics (Census ACS)
  • Travel times (Google Distance Matrix API)
  • Geographic boundaries (Census TIGER/Line)

All base counts (employment, payroll, households, population) are directly from official government APIs. No synthetic or randomly generated data is used.

12.2 What is Estimated?

Statistical Estimates (Based on Real Data):
  • Employment by Industry: Census Bureau suppresses detailed employment for privacy. We estimate proportionally: Emp_industry = Emp_total × (Estab_industry / Estab_total)
  • Revenue: Not available from Census. Estimated using BLS revenue-per-employee ratios: Revenue = Employment × Revenue_per_Employee (BLS)
  • Corporate Score: Calculated metric combining real and estimated data
  • Geometric Score: Calculated metric from real data

12.3 Limitations

  • Revenue Estimates: Based on national industry averages from BLS/Census benchmarks. The simplified model uses $200K/employee for general industries and $350K/employee for power industries. Actual revenue varies significantly within sectors and by geographic location. Local variations (cost of living, sector mix nuances, business size distribution) are not captured. A ZIP code with many small businesses may have lower actual revenue than estimated, while one with large corporate headquarters may have higher revenue.
  • Employment by Industry: Proportional estimation assumes establishments are similar in size
  • Travel Times: Reflect conditions at time of API call, may vary by time of day
  • ZIP Code Boundaries: ZCTAs may not perfectly match actual ZIP code delivery areas
  • Geographic Coverage: Analysis limited to 7 major metropolitan areas
  • Distance/Time Component: The assumption that "farther from airport = better" is a normative model choice. Sensitivity analysis could test alternative weightings (e.g., reducing from 20% to 10%) or functional forms (e.g., U-shaped function favoring moderate distances). Results may be robust to these variations, but this has not been tested.

12.4 Data Quality Assurance

  • All data sources are official U.S. government agencies
  • No synthetic or randomly generated data is used
  • All estimates are based on established statistical methodologies
  • Data is cached to ensure reproducibility
  • Validation checks are performed to identify outliers and errors
